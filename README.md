# LinkedIn Post Studio — Technical Specification & Implementation Blueprint

A self-serve tool that helps an **individual professional** maintain a consistent LinkedIn
presence with low effort, while keeping **full editorial control**. The system discovers
topics from the user's expertise, generates a daily post (caption + image + formatting),
stores it for review, emails a daily preview, and publishes **only** after the user reviews
and clicks **Publish**. There is **no auto-publishing** — a human review step is mandatory by design.

---

## Who this document is for

- **Developers** — everything needed to start building: architecture, schema (SQL DDL),
  module specs, integration contracts, and a phased roadmap.
- **Non-technical stakeholders** — each section opens with a plain-language summary
  ("In plain terms") so you can follow how the pieces fit without reading code.

## The one-paragraph version

A daily background job looks at what the user is good at (their saved skills, niche, and a
sample of their own past posts that *they* provide), picks a topic, and asks an AI model to
write a LinkedIn-ready caption plus generate a matching image. The draft lands in a database
and the user gets an email: *"Today's post is ready."* They open a dashboard, edit anything
they want, and press **Publish**. Only then does the post go to LinkedIn through LinkedIn's
official API. Everything is logged.

## ⚠️ Three buildability flags to read first

These are real-world constraints that shape the whole design. Full detail in
[Assumptions & Open Questions](docs/09-assumptions-open-questions.md) and
[Integrations](docs/06-integrations.md).

1. **Reading the user's historical LinkedIn posts via API is NOT self-serve available.**
   LinkedIn does not grant individual/self-serve apps a scope to read a member's feed or
   post history. Scraping it would violate LinkedIn's Terms. **Our compliant approach:** the
   user manually pastes a few representative posts during onboarding, and/or uploads their
   own LinkedIn data export (LinkedIn lets every member download their own archive). This is
   the single biggest deviation from a naive reading of the brief — and it is non-negotiable
   for compliance.

2. **Posting to a personal profile IS supported** via LinkedIn's official **"Share on
   LinkedIn"** product (`w_member_social` scope) + the **Posts API**. This is the green path
   and the design relies on it. It still requires creating a LinkedIn Developer app and
   getting the product approved.

3. **No paid APIs — the AI is self-hosted and free.** Captions are generated by a local
   **Ollama** model (Llama 3.1 / Qwen2.5) and images by self-hosted **Stable Diffusion
   (SDXL via ComfyUI)**. Email goes out over free **SMTP (Nodemailer)**, jobs run on
   **pg-boss** (a queue inside Postgres), and storage is **MinIO/local disk** — the whole
   stack runs in Docker on your own hardware. The only external dependency is **LinkedIn's
   API, which is free to call**. (Midjourney is excluded — Discord-only, no API.) This
   requires a server/box to run inference; a keyless free fallback exists for machines
   without a GPU — see [Tech Stack §5.2](docs/05-tech-stack.md#52-the-no-api-cost-principle--how-ai-runs-locally).

## Core design principles

- **Human-in-the-loop is a hard constraint**, not a setting. The publish path cannot be
  reached without an explicit user action on a reviewed draft.
- **Tokens never touch the browser.** All LinkedIn credentials live encrypted server-side.
- **Provider-agnostic AI.** Caption LLM and image generator sit behind interfaces so they
  can be swapped without touching business logic.
- **Built for one, designed for many.** The data model and job system are multi-tenant from
  day one even though the target user manages a single account.
- **Everything is logged and reversible.** Generation, edits, emails, and publishes are
  auditable.
